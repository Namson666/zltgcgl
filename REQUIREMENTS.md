# 资料通工程管理系统 — 需求文档

> 施工企业多租户 SaaS 管理平台，整合物资管理（WMS）和劳资管理两大核心模块。
> 本文档仅供需求参考，不含任何代码实现细节。

---

## 一、用户角色

| 角色 | 职责 |
|------|------|
| **平台开发者** | 超级管理员，管理所有租户、平台配置、系统监控 |
| **企业管理员** | 租户内最高权限，拥有全部功能 |
| **企业普通用户** | 按角色分配功能和数据权限 |

## 二、认证与权限

### 2.1 登录方式
- **开发者登录**：用户名 + 密码
- **企业用户登录**：企业代码 + 用户名 + 密码
- **用户自助注册**：创建企业 → 开通试用订阅 → 自动创建管理员账号

### 2.2 会话管理
- JWT 短期令牌（默认 7 天有效）
- 长期凭证（30 天有效），用于自动续期，用完即焚
- 开发者可临时"模拟"企业用户身份（4 小时有效），方便排查问题

### 2.3 权限控制
- 基于角色的访问控制（RBAC），18 个细粒度功能权限：
  - 看板、系统管理、日志、导出、库存查看
  - 入库、出库、退库、调拨
  - 记录查看、班组台账
  - 人员管理、考勤管理、工资核算、工资发放、风控管理、报表
  - 合同管理、项目部管理
- 数据范围控制：可限制用户只能看自己所属项目部的数据

## 三、物资管理（WMS）

### 3.1 物资目录
- 物资基本信息：编码、名称、规格、单位、单价、分类
- 支持增删改查、Excel 导出

### 3.2 送货单 + OCR 智能识别
- 上传送货单图片（JPG/PNG）或 PDF
- 自动 OCR 识别文字内容
- AI 自动提取：供应商名称、合同编号、送货单号、送货日期、物资明细清单
- 识别结果自动匹配系统中已有的供应商、合同、项目
- PDF 多页自动逐页识别，方向异常自动纠正
- 确认后一键转入库单

### 3.3 入库管理
- **手动录入**：选择合同 → 项目部 → 项目名称（三级联动），填写物资明细
- **Excel 批量导入**：上传 Excel 自动创建物资并入库
- **送货单转入库**：OCR 识别后的送货单可直接生成入库单
- 入库单支持编辑、删除（自动调整库存）
- 入库时自动更新库存

### 3.4 库存查询
- 多维度筛选：合同、项目部、项目名称、物资名称/编码、班组
- 状态标识：正常库存、待分配库存
- Excel 导出

### 3.5 出库管理
- 选择项目部 → 选择项目名称 → 勾选有库存的物资 → 填写数量和领用班组
- 出库单自动生成 PDF（中文，含表格和签收栏）
- 出库单删除时自动恢复库存
- Excel 导出

### 3.6 退库管理
- 选择历史出库记录进行退库
- 支持 Excel 批量导入退库
- 退库后自动恢复库存

### 3.7 物资调拨
- 从 A 项目部/项目名称调拨物资到 B 项目部/项目名称
- 调拨 = 来源库存减少 + 目标库存增加
- 支持借调归还

### 3.8 班组台账
- 按班组查询物资领用明细
- 按班组、项目部、日期筛选
- Excel 导出

### 3.9 供应商管理
- 供应商基本信息维护
- 与合同关联

### 3.11 班组管理
- 班组信息：名称、组长、电话、人数

## 四、劳资管理

### 4.1 人员管理
- 人员基本信息：姓名、身份证号、手机、人员类型（工人/职员）、项目部、分包商
- 薪资信息：月薪制/日薪制、社保
- 入职、离职、复职流程
- 自动检测：未成年、超龄、身份证重复

### 4.2 考勤管理
- 每日考勤：出勤/半勤/缺勤 + 加班（2/4/6/8小时）
- 支持批量录入
- 每人每天唯一一条考勤记录
- 月度考勤汇总

### 4.3 工资核算
- 批量自动化核算：选择月份 + 项目部 + 人员范围
- 计算公式：
  - 应付工资 = 出勤天数 × 日薪
  - 缺勤扣款 = 缺勤天数 × 日薪
  - 加班费 = 加班天数 × 日薪
  - 社保扣款
  - 实发 = 应付 - 缺勤 - 社保 + 加班 + 自定义加项 - 自定义扣项
- 支持手动调整加班费、自定义加减项
- 欠薪追踪：应发 - 已发
- 考勤变更后自动标记"需重新核算"

### 4.4 工资发放
- 录入发放记录：收款人、身份证号、金额、发放日期、发放方式
- 支持未在人员名册中的收款人（如临时工）
- 批量确认发放
- AI 辅助匹配发放记录到人员身份
- 欠薪抵扣

### 4.5 分包商管理
- 分包商类型：公司 / 个人
- 基本信息：名称、联系人、电话、身份证号、银行账号

### 4.6 分包合同管理
- 关联总包合同 + 分包商
- 合同金额管理

### 4.7 产值管理
- 按月录入产值 + 应付比例
- 分包进度款支付（可关联多笔产值）

### 4.8 风控异常
- 系统自动标记异常（黄色/红色两级）
- 异常处理（需填写处理原因）
- 风控统计面板

### 4.9 报表导出
- 综合报表：月度工资汇总 + 欠薪统计 + 异常人员明细 + 发放记录
- 按月份、项目部筛选

### 4.10 附件管理
- 文件上传（支持多文件，单文件上限 50MB）
- 关联到任何业务实体（如人员档案、合同等）

## 八、核心业务规则

### 8.1 数据隔离（三级层级）
```
合同 → 项目部 → 项目名称
```
- 合同是顶层容器，一个合同可有多个项目部
- 项目部是中间层，一个项目部可有多个子项目
- 项目名称是最细粒度，物资库存必须按项目名称隔离
- 不同项目名称的库存不能混淆

### 8.2 库存管理规则
- 每个库存记录由「项目部 + 物资 + 项目名称」三者唯一确定
- 项目名称为空 = "待分配库存"
- 入库：存在则累加，不存在则新建
- 出库：从指定项目名称的库存中扣减
- 退库：恢复到原项目名称的库存
- 调拨：来源减 + 目标增
- 删除入库单：自动扣回库存
- 删除出库单：自动恢复库存

### 8.3 OCR 识别流程
```
上传文件
  ├── PDF → 转图片 → 逐页 OCR 识别 → 方向异常自动纠正
  └── 图片 → 直接 OCR 识别
→ AI 结构化提取 → 物资清单、供应商、送货单号
→ 匹配系统已有数据 → 转送货单 → 分组生成入库单
```

### 8.4 软删除
- 关键业务数据（合同、入库单、出库单、租户）采用软删除
- 删除后数据保留，可恢复
- 恢复时关联数据联动恢复（如恢复入库单同时恢复库存）

### 8.5 多租户
- 所有业务数据必须带企业标识
- 用户登录后只能访问本企业数据
- 开发者可通过"企业视角"临时切换到某企业

### 十三、OCR 识别与数据填入完整流程

#### 13.1 整体链路

```
用户上传图片/PDF 
  → 前端 FormData POST 到 /api/wms/delivery-orders/ocr（超时10分钟）
  → 后端两阶段处理：
      阶段1：百度 OCR（图片→文本行）
      阶段2：AI 大模型结构化（文本→JSON）
  → 返回给前端：识别文本 + 结构化数据 + 系统匹配结果
  → 前端自动填充表单，用户校验确认后提交手动入库
```

#### 13.2 阶段1：百度 OCR 识别

**步骤 A — 读取 OCR 配置**
- 从数据库 `OcrConfig` 表查找 `provider='baidu'` 且 `isEnabled=true` 的配置
- 如无启用配置，回退查找任何百度配置
- 优先取 `apiKey` 字段，兼容 `secretId` 字段
- 无有效配置时报错：「请先在开发者后台的OCR配置中配置」

**步骤 B — 文件类型判断**
- **PDF 文件**：
  1. 用 Ghostscript（`gs` 命令）将 PDF 逐页转为 PNG 图片（150 DPI）
  2. 逐页发送百度 OCR 识别
  3. 每页检测中文有效占比，若乱码率 > 60% 且行数 > 10，自动尝试用 Python PIL 旋转 180 度重新识别
  4. 每页结果添加 `=== 第N页（共M页）===` 分隔标记
- **图片文件**（JPG/PNG）：
  1. 直接读文件转 base64
  2. 调用百度 OCR single call

**步骤 C — 百度 OCR API 两步调用**
1. 第一步：POST `https://aip.baidubce.com/oauth/2.0/token`，传 `client_id` + `client_secret`，获取 `access_token`
2. 第二步：POST `https://aip.baidubce.com/rest/2.0/ocr/v1/accurate_basic?access_token=xxx`，Body 传 `image=<base64>&language_type=CHN_ENG&detect_direction=true`
3. 返回格式：`{words_result: [{words: "每行文字"}], words_result_num: 行数}`，最终提取为字符串数组

**使用的 OCR 接口**：百度 accurate_basic（免费高精度版），已验证可识别 45+ 条物资的送货单

#### 13.3 阶段2：AI 大模型结构化提取

**输入**：百度 OCR 返回的全部文本行，用换行符 `\n` 拼接成一个长字符串

**AI 提供商路由**：
- 从 `AiConfig` 表查找 `isEnabled=true` 的最新配置
- 支持 `openai` 和 `minimax` 两种提供商
- 请求格式：
  ```json
  {
    "model": "gpt-4o-mini 或 MiniMax-Text-01",
    "messages": [
      {"role": "system", "content": "<系统提示词>"},
      {"role": "user", "content": "<OCR原始文本>"}
    ],
    "temperature": 0.05,
    "max_tokens": 32768
  }
  ```

**系统提示词设计要点**：
1. 角色：专业的物资管理系统数据提取助手，处理电力工程企业的送货单
2. 任务：从 OCR 文字中精确提取结构化数据，返回 JSON
3. 过滤规则：只提取带序号（1,2,3...）的物资明细行；忽略小计行、合计行、签字行、备注段落、空行；同种物资不同序号分别提取，不合并去重
4. 字段提取规则：
   - `supplier`：从"供应商（盖章）："等关键词后提取公司名称
   - `deliveryNo`：从"送货通知单号："后提取
   - `deliveryDate`：YYYY-MM-DD 格式
   - `items[].name`：完整保留物资名称
   - `items[].unit`：计量单位
   - `items[].quantity`：只保留数字和小数点
   - `items[].projectName`：完整拼接分布在不同行的项目名称（不截断、不省略）
5. 约束：不编造数据，不确定的填空字符串或 0，不能改变顺序

**响应解析策略**（多重兜底）：
1. 剥离 `<think>...</think>` 标签（处理推理模型的思考过程）
2. 按优先级匹配 JSON：markdown 代码块 → 完整 JSON 对象 → 片段 JSON
3. 数量清洗：`rawQty.replace(/[^\d.]/g, '')` 只保留数字和小数点
4. 容错：如 items 为空但有 materialName，生成单条记录

**AI 返回的 JSON 结构**：
```json
{
  "supplier": "萍乡市中源瓷业有限公司",
  "deliveryNo": "HYRN20250501",
  "deliveryDate": "2025-05-01",
  "projectName": "2025技改/遵义播州供电局",
  "items": [
    {
      "name": "绝缘子",
      "unit": "片",
      "quantity": 150,
      "projectName": "2025技改/遵义播州供电局"
    }
  ]
}
```

#### 13.4 系统自动匹配

在 AI 返回结果后，后端自动匹配系统已有数据：

| 匹配项 | 逻辑 | 返回 |
|--------|------|------|
| 供应商匹配 | 用 `parsed.supplier` 前10字符模糊查 `Supplier` 表 | `{id, name, isNew}` |
| 合同匹配 | 用 `parsed.contractCode` 模糊查 `Contract.code` | `{id, name}` 或 null |
| 项目匹配 | 取 `parsed.projectName` 或首条 item 的 projectName 查 `SubProject.name` | `{id, name}` 或 null |

#### 13.5 返回前端的完整结构

```json
{
  "success": true,
  "data": {
    "rawText": "百度OCR识别的全部原始文本",
    "parsed": {
      "supplier": "供应商名称",
```

#### 13.6 前端数据填入与用户确认

1. **前端收到结果后自动操作**：
   - 将 `parsed.supplier` 填入供应商输入框，并锁定（防止误改）
   - 将 `parsed.deliveryDate` 填入日期选择器
   - 将 `parsed.items` 逐条填入物资明细表格，每条包含：物资名称、单位、送货数量、实收数量、单价、项目名称
   - 自动切换到「手动录入」tab 供用户校验

2. **用户校验操作**：
   - 点击「修改」按钮解锁供应商字段
   - 逐条检查物资明细，可修改名称、单位、数量、单价、项目名称
   - 可手动添加遗漏行或删除错误行
   - 选择合同 → 项目部联动下拉（项目部按所选合同过滤）
   - 项目名称输入时自动显示已有项目名称的下拉建议（datalist）

3. **用户确认提交** → 调用 `POST /api/wms/inbound/manual`，数据进入手动入库流程

#### 13.7 手动入库后端处理

1. 接收数据：`{subProjectId, contractId, supplierName, inboundDate, items: [{materialName, unit, quantity, unitPrice, projectName}]}`
2. 逐条处理物资：
   - 按 `materialName` 查 `Material` 表
   - **查不到则自动创建**（名称、单位、单价）
   - 查到则使用已有 ID
3. 创建入库单（`InboundOrder`），状态为 `source='manual'`
4. 写入库存（`Inventory.upsert`）：
   - 唯一键：`subProjectId + materialId + projectName`
   - 存在则累加数量，不存在则新建
   - `projectName=null` 表示「待分配库存」

#### 13.8 送货单转入库（独立功能，非 OCR 路径）

当前 OCR 识别结果填入手动入库表单；此外还有一个独立的「送货单转入库」功能：

1. 从 `DeliveryOrder` 表获取送货单及其明细
2. **按项目名称分组**（`projectCode` 优先，其次 `projectName`，兜底 `_default`）
3. 每组生成一个入库单：
   - 确定子项目 ID（按 code 或 name 模糊匹配 `SubProject` 表）
   - 逐条明细：查物资（按 code）→ 找不到则自动创建 → 组装入库明细
   - 创建入库单（`source='delivery'`）
   - 写入库存 `Inventory.upsert`

#### 14.1 权限配置

老板、财务总监、超级管理员拥有所有合同及以下子目录劳资、物资、合同的权限
项目经理可以拥有多个项目的及以下子目录劳资、物资的权限
项目劳资员可以拥有多个项目的及以下子目录劳资的权限
项目材料员可以拥有多个项目的及以下子目录物资的权限
总部劳资员可以拥有所有合同及以下子目录劳资的权限
总部材料员可以拥有所有合同及以下子目录物资的权限
所有板块可以按级自定义角色和自定义权限
可以按层级给批量权限，总部层级=所有合同权限、所有劳资权限、所有物资权限，项目层级=可以选择多个项目部合同权限、劳资权限、物资权限
以上权限是读取、修改、删除权限。但所有操作有操作日志，操作日志只能超级管理员可以删除。
