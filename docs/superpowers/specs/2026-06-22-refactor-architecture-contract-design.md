# 资料通工程管理系统重构架构合同

日期：2026-06-22  
状态：设计草案，待用户审阅确认  
范围：模块开通、租户登录门户、合同基础板块、人员人脸与小程序打卡、后续重构顺序

## 1. 设计目标

本次重构不是单点修复，而是把现有系统整理成可长期演进的模块化 SaaS 架构。

目标：

- 保留现有工程管理系统主线能力。
- 明确默认基础功能和可开通业务模块的边界。
- 让企业只看到并使用已开通的业务模块。
- 合同管理作为默认基础功能，支撑物资、劳资、财务三条业务线。
- 支持企业独立登录页、独立域名、品牌 logo 和公司名称配置。
- 为后续微信小程序打卡、人脸资料、定位照片打卡留好接口和数据结构。

## 2. 需求来源分类

### user_stated

用户明确提出：

- 物资管理、劳资管理、财务管理三个板块需要由开发者后台给企业开通。
- 未开通的板块不显示，也不影响已开通板块正常使用。
- 每个企业可以有独立登录界面，可配置独立域名、登录页 logo、公司名字。
- 不论是否设置独立登录界面，原企业代码登录系统必须保留。
- 合同管理是默认基本功能，不作为业务开通模块。
- 合同管理下面需要承包合同、采购合同、分包合同三个 tab。
- 承包合同参考原合同管理，并增加合同上传下载、收款记录、附件集中管理。
- 采购合同可关联承包合同，支持合同上传下载、发票、附件、支付记录、付款凭证。
- 分包合同可关联承包合同，可关联班组，支持付款记录、付款凭证、结算凭证。
- 人员管理增加上传照片录入人脸。
- 微信小程序通过人员手机号登录，当天拍照采集定位和照片完成打卡。
- 微信打卡必须关联到该人员。
- 每天打卡一次或两次由公司设置，不限具体打卡时间，不限打卡位置。
- 打卡必须返回具体位置；离开平时打卡所在县份时，记录为异常打卡。
- 异常不能一直异常，系统需要支持批量处理异常。
- 可以将异常地点添加为信任打卡地；信任打卡地只针对个人，不影响其他人员。
- 打卡照片必须存档，并需要做人脸识别。
- 采购合同和分包合同不用审批。
- 合同附件可以下载，也可以删除。
- 企业独立登录页不需要企业代码登录。
- 小程序接入由开发者管理；企业可以接自己的小程序，未接入时默认使用开发者小程序。

### ai_inferred

基于现有代码和需求合理推断：

- 合同管理是物资、劳资、财务的基础数据入口。
- 模块开通必须在前端菜单、前端路由、后端 API 三层同时生效，不能只靠隐藏菜单。
- 用户权限需要和企业模块开通叠加判断：企业未开通模块时，即使用户有权限，也不能访问模块。
- 合同附件、发票、付款凭证、结算凭证应共用统一附件能力，避免每个模块各写一套上传逻辑。
- 小程序打卡应先支持照片、定位、时间、人员手机号的可审计记录，再决定是否接入真正人脸识别算法。
- 小程序打卡需要支持多租户数据分流：企业自有小程序通过 appId 直接识别企业；开发者默认小程序需要通过手机号匹配人员，并在冲突时消歧。
- 人脸识别建议设计成可替换服务适配层，便于后续选择腾讯云、百度云、阿里云或其他供应商。
- 由于当前后端 build 失败，正式重构前必须先恢复构建基线。

### needs_user_confirmation

后续仍需用户确认：

- 人脸识别供应商选择：腾讯云、百度云、阿里云，或后续自建。当前先按可替换适配器设计。
- 默认开发者小程序中，如果手机号匹配到多个企业，是否允许用户选择公司，还是必须由管理员预绑定。
- 异常打卡的处理流程：仅记录异常，还是需要管理员确认。

当前设计不立即包含：

- 复杂审批流引擎。
- 电子签章。
- 多组织或集团级跨企业合同汇总。

## 3. 模块边界

### 默认基础功能

默认所有企业都具备：

- 登录 / 企业代码登录
- 企业基础信息
- 用户管理
- 角色权限
- 项目部管理
- 合同管理
- 附件上传下载基础能力
- 基础操作日志
- 基础看板入口

合同管理属于默认基础功能，不由模块开通控制。

### 可开通业务模块

由开发者后台为企业开通：

- 物资管理：wms
- 劳资管理：labor
- 财务管理：finance

企业未开通某模块时：

- 前端菜单不显示。
- 前端路由不可进入。
- 后端 API 返回模块未开通错误。
- 模块内权限不生效。
- 不影响其他已开通模块使用。

## 4. 后端公共接口与调用规则

### 模块开通检查

新增统一检查逻辑：

请求进入 API -> authenticate -> requireUser -> requireTenantModule(moduleKey) -> requirePermission(permissionKey) -> route handler

合同管理不走 requireTenantModule，但仍走登录和权限检查。

### 用户权限判断

权限判断应拆成三层：

- 企业层：企业是否开通业务模块。
- 用户层：用户角色是否拥有具体功能权限。
- 数据层：用户可访问哪些项目部、合同、项目名称。

### 数据范围判断

统一抽象 UserScope：

- tenantId：所有业务数据必须归属租户。
- departmentIds：用户可访问的项目部。
- contractIds：由项目部或显式授权推导。
- projectName：物资库存细粒度隔离字段。

## 5. 数据来源与字段主线

### 租户主线

Tenant 是企业数据根节点。

所有业务表必须包含或可追溯到：

- tenantId
- isActive 或删除状态
- 创建 / 更新时间

### 合同主线

合同作为基础板块，应统一承载三类合同：

- construction：承包合同
- purchase：采购合同
- subcontract：分包合同

建议保留现有 Contract，逐步演进，而不是一次性推翻。

核心字段：

- tenantId
- contractType
- parentContractId：采购合同、分包合同可关联承包合同
- supplierId / subcontractorId
- workTeamId 或合同-班组关联表
- totalAmount
- startDate
- endDate
- status

### 附件主线

合同文件、发票、付款凭证、结算凭证不应散落在多个表中。

建议统一 ContractAttachment：

- tenantId
- contractId
- attachmentType：contract_file、invoice、payment_voucher、receipt_voucher、settlement_voucher、other
- fileName
- fileUrl
- fileSize
- mimeType
- uploadedBy
- createdAt

### 物资主线

物资模块依赖合同与项目部，但只有企业开通 wms 后才显示和可访问。

库存唯一主线：

tenantId + departmentId/subProjectId + materialId + projectName

入库、出库、退库、调拨必须通过统一库存变更服务，不允许各自直接随意修改库存。

### 劳资主线

劳资模块依赖人员、项目部、班组、分包合同。

核心关系：

- Personnel
- AttendanceRecord
- SalaryRecord
- PaymentRecord
- Anomaly
- PersonnelFaceProfile
- AttendanceCheckIn

人员手机号在小程序登录场景中是关键识别字段，但不应跨租户全局唯一；同一企业内应尽量唯一。

### 财务主线

财务模块应围绕合同和凭证汇总：

- 承包合同：收款记录、收款凭证。
- 采购合同：支付记录、付款凭证、发票。
- 分包合同：付款记录、付款凭证、结算凭证。
- 费用报销：项目部、合同、备用金账户。
- 合同盈亏：聚合合同收款、付款、费用、工资、分包支出。

## 6. 页面运行流

### 开发者后台：企业模块开通

开发者进入租户管理 -> 选择企业 -> 打开模块开关 -> 保存 -> 后端写入 TenantModuleEntitlement -> 企业用户重新进入系统 -> 前端读取当前企业已开通模块 -> 菜单和路由按模块显示 -> 后端 API 同步拦截未开通模块。

### 企业登录页

原登录方式：

访问统一登录页 -> 输入企业代码、用户名、密码 -> 后端根据企业代码找到 Tenant -> 校验用户 -> 签发 token -> 进入企业后台。

独立登录页：

访问企业独立域名 -> 前端请求 portal config -> 展示企业 logo、公司名、登录页样式 -> 用户输入用户名、密码 -> 后端根据域名识别 Tenant -> 校验用户 -> 签发 token -> 进入企业后台。

独立登录页不需要企业代码，因为域名已经完成企业识别。原统一登录页仍保留企业代码登录。

### 合同管理

进入合同管理 -> 默认显示承包合同 tab -> 用户可切换采购合同 / 分包合同 -> 新增或编辑合同 -> 上传合同文件、附件、凭证 -> 承包合同维护收款记录 -> 采购合同维护发票、支付记录、付款凭证 -> 分包合同维护班组关联、付款记录、结算凭证。

### 小程序打卡

人员打开微信小程序 -> 系统识别小程序来源 -> 输入手机号 -> 系统定位企业和人员 -> 进行人脸比对 -> 获取定位并反查县份 -> 拍照上传存档 -> 生成 AttendanceCheckIn -> 异常打卡写入异常记录 -> 后台劳资模块查看打卡记录 -> 后续可关联 AttendanceRecord。

## 7. 小程序多企业接入与数据分流

系统支持两种小程序接入模式：

1. 企业自有小程序
   - 开发者后台为企业配置该企业自己的 appId、appSecret。
   - 请求进入后，系统通过 appId 直接识别 tenantId。
   - 手机号只在该 tenantId 内匹配人员。

2. 开发者默认小程序
   - 未接入自有小程序的企业，默认使用开发者的小程序。
   - 默认小程序服务多个企业，因此不能只用手机号直接确定企业。
   - 手机号匹配时需要先找候选企业，再消除冲突。

推荐数据分流规则：

1. 后端读取小程序 appId。
2. 如果 appId 已绑定某企业，则直接得到 tenantId。
3. 如果 appId 是开发者默认小程序，则按手机号查找候选人员。
4. 如果手机号只命中一个企业，则绑定该企业和人员。
5. 如果手机号命中多个企业，则返回候选企业列表，让用户选择公司，或由后台管理员预绑定。
6. 选择或绑定完成后，后端创建 MiniProgramSession，里面固定 tenantId 和 personnelId。
7. 后续打卡必须使用 session 中的 tenantId 和 personnelId，不再每次全局猜测。

防冲突原则：

- 手机号不能跨全平台唯一，因为同一个工人可能在不同企业档案中出现。
- 同一企业内手机号建议唯一。
- 打卡记录必须保存 tenantId、personnelId、phone、miniProgramAppId。
- 默认小程序场景下，如果手机号命中多个企业且未选择公司，不允许直接打卡。

建议新增 MiniProgramAppConfig：

- tenantId：为空表示开发者默认小程序；不为空表示企业自有小程序。
- appId
- appSecretEncrypted
- appName
- mode：developer_default 或 tenant_owned
- isEnabled
- createdAt
- updatedAt

开发者后台负责管理这些配置。

## 8. 打卡规则与异常判断

建议新增 AttendancePolicy：

- tenantId
- checkInsPerDay：1 或 2
- requireFaceRecognition：默认 true
- requireLocation：默认 true
- countyAnomalyEnabled：默认 true

用户已确认：

- 每天打卡一次或两次由公司设置。
- 不限具体打卡时间。
- 不限具体打卡位置。
- 必须反馈具体位置。
- 离开平时打卡所在县份，按异常打卡记录。
- 异常可以批量处理。
- 异常地点可以加入个人信任打卡地，加入后该人员在该地点或县份打卡不再记为异常。

系统需要把经纬度反查到县或区级行政区。

建议规则：

1. 每个人员保存 usualCountyCode 和 usualCountyName。
2. usualCounty 可以来自项目部所在地，也可以由第一次正常打卡后管理员确认。
3. 每个人员可以维护多个 TrustedCheckInLocation。
4. 每次打卡根据经纬度反查 currentCountyCode。
5. 如果 currentCountyCode 与 usualCountyCode 不一致，先检查是否命中该人员的信任打卡地。
6. 命中信任打卡地则正常记录。
7. 未命中信任打卡地则标记异常。
8. 异常不阻止打卡，只记录异常原因。

异常字段建议：

- isAbnormal
- abnormalReason：out_of_usual_county、location_missing、face_mismatch 等。
- currentCountyCode
- currentCountyName
- usualCountyCode
- usualCountyName
- trustedLocationId：命中个人信任打卡地时记录

### 个人信任打卡地

建议新增 TrustedCheckInLocation：

- tenantId
- personnelId
- countyCode
- countyName
- locationName
- latitude
- longitude
- radiusMeters：可选，默认按县份信任，也可后续支持更细范围
- sourceCheckInId：从哪条异常打卡转为信任地点
- createdBy
- createdAt
- isActive

信任打卡地规则：

- 只对该 personnelId 生效。
- 不影响同公司其他人员。
- 可以从异常打卡记录一键添加。
- 可以批量把多条异常标记为已处理。
- 已处理异常保留原始异常记录，不物理删除。

### 异常批量处理

异常处理建议支持：

- 批量标记为已处理。
- 批量添加处理备注。
- 从单条异常打卡添加为个人信任打卡地。
- 批量忽略但保留记录。
- 查看处理人、处理时间、处理原因。

异常处理状态建议：

- pending：待处理。
- accepted：已接受，作为正常情况留档。
- trusted_added：已添加为个人信任打卡地。
- rejected：确认异常。

## 9. 人脸识别建议

用户要求打卡照片要通过识别，但不确定是否需要第三方。

建议设计：

- 系统先做 FaceRecognitionProvider 适配层。
- 第一版建议接第三方云服务做人脸比对，因为自研识别模型成本高、准确率和合规风险都更高。
- 人员管理上传的人脸照片作为注册照。
- 打卡照片作为现场照。
- 后端调用人脸比对，得到相似度和结果。
- 低于阈值则标记异常，但保留打卡记录。

推荐数据流：

人员档案上传照片 -> 生成 PersonnelFaceProfile -> 小程序打卡上传现场照 -> FaceRecognitionProvider 比对注册照和现场照 -> 写入 AttendanceCheckIn.faceMatchScore 和 faceMatchStatus。

识别结果：

- passed：识别通过。
- failed：识别失败。
- manual_review：需要人工复核。
- provider_error：识别服务异常，但照片和定位仍存档。

系统不应因为第三方识别服务短暂不可用就丢失打卡照片和定位。

## 10. 合同附件补充规则

用户已确认：

- 采购合同和分包合同不用审批。
- 合同附件可以下载。
- 合同附件可以删除。

建议删除采用软删除：

- 前端显示删除。
- 后端标记 isDeleted 或 deletedAt。
- 文件是否物理删除可后续配置。
- 操作日志记录删除人、删除时间、合同 ID、附件 ID。

## 11. 替换/删除影响

### 如果删除模块开通机制

影响：

- 无法按企业套餐控制物资、劳资、财务。
- 前端隐藏和后端权限会继续分裂。
- SaaS 商业化套餐无法落地。

### 如果合同不作为基础功能

影响：

- 物资、劳资、财务都需要处理合同缺失逻辑。
- 采购合同、分包合同、财务收付款会变得碎片化。
- 用户体验会复杂化。

因此合同管理必须作为默认基础功能。

### 如果不统一附件能力

影响：

- 合同文件、发票、付款凭证、结算凭证会各自实现上传下载。
- 权限、文件删除、文件迁移、存储路径会难以维护。

### 如果小程序打卡直接做人脸算法

影响：

- 成本、准确率、隐私合规、第三方供应商选择都会提前复杂化。
- 建议第一版先做照片定位打卡闭环，再升级为人脸比对。

## 12. 第一阶段实施边界

第一阶段只做设计和工程基线，不直接完整实现全部功能。

建议第一阶段目标：

1. 修复后端 TypeScript build。
2. 建立本地验证脚本。
3. 增加模块开通数据结构和后端 guard。
4. 前端菜单根据模块开通状态显示/隐藏。
5. 合同管理重构前先建立字段契约和附件契约。

不在第一阶段做：

- 小程序前端。
- 人脸识别算法。
- 全量合同 UI 重写。
- 大规模数据库迁移。
- WMS/Labor/Finance 全量拆分。

## 13. 推荐重构顺序

1. 工程稳定：修 build、验证脚本、测试状态。
2. 模块开通底座：TenantModuleEntitlement + 后端 guard + 前端菜单控制。
3. 租户登录门户：TenantPortalConfig + 域名识别 + 品牌配置。
4. 合同基础板块重构：三 tab + 附件/收款/付款/发票/凭证。
5. 小程序接入配置：开发者默认小程序 + 企业自有小程序。
6. 劳资人员人脸资料：注册照、人员手机号约束、人脸服务适配层。
7. 小程序打卡接口：手机号识别、企业消歧、定位县份、照片、人脸比对、异常记录。
8. WMS/Labor/Finance 内部模块瘦身。

## 14. 验收标准

设计阶段验收：

- 用户确认默认基础功能和可开通模块边界。
- 用户确认合同三 tab 的业务范围。
- 用户确认小程序打卡规则：每天 1 次或 2 次由企业配置，不限时间，不限位置，跨常用县份记异常。
- 用户确认默认开发者小程序和企业自有小程序两种接入模式。

工程阶段验收：

- 后端 build 通过。
- 前端 build 通过。
- 前后端测试通过。
- 未开通模块前端不可见。
- 未开通模块后端 API 不可访问。
- 合同管理默认可见。
- 原企业代码登录继续可用。
- 独立域名登录不影响原登录方式。
- 企业独立登录页不需要企业代码。
- 企业自有小程序按 appId 直接分流到企业。
- 默认开发者小程序可通过手机号匹配人员，冲突时不允许静默打卡。
- 打卡记录关联 tenantId 和 personnelId。
- 打卡记录保存照片、定位、县份、人脸识别结果和异常状态。
- 异常打卡可以批量处理。
- 个人信任打卡地只对对应人员生效。
