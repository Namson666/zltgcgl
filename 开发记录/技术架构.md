## 四、核心架构设计

### 4.1 三级数据隔离

```
合同(Contract) → 项目部(Department) → 项目名称(projectName)
    一级              二级                三级
```

- 一个合同可有多个项目部，一个项目部可有多个项目名称
- 每层之间数据隔离，跨项目部/项目名称的库存不能混淆
- 通过 Prisma 中间件 + 路由中间件双重过滤实现

### 4.2 多租户架构

采用 **Developer → Tenant** 模型：
- Developer（平台开发者/超级管理员）管理所有租户
- Tenant（施工企业）拥有独立的数据空间
- 所有业务表必须包含 `tenantId` 字段
- 数据隔离通过 Prisma 中间件自动注入 `tenantId` 条件

### 4.3 权限体系

RBAC + 数据范围双重控制：
- **RBAC**：动态 Role + Permission 表（18 个细粒度功能权限）
- **数据范围**：ALL（全企业）/ DEPARTMENTS（授权项目部列表）/ OWN_DEPARTMENT（所属项目部）
- 默认 8 种角色：超级管理员、老板、管理员、物资员、劳资员、财务、出纳、项目出纳

### 4.4 库存管理规则

- 库存唯一键：`subProjectId + materialId + projectName`
- `projectName = null` 表示「待分配库存」
- 入库时 upsert（存在累加，不存在新建）
- 出库/退库/调拨均通过 `findFirst` + 条件 `update` 安全操作
- 软删除（`isActive: false`）确保数据可恢复

### 4.5 OCR 识别流水线

```
文件上传 → 类型判断 → PDF转PNG (Ghostscript) → 百度OCR识别
→ AI 大模型结构化提取 (MiniMax/OpenAI) → 系统数据自动匹配
→ 前端自动填充表单 → 用户校验确认 → 入库
```

## 五、关键技术决策

| 决策 | 原因 |
|------|------|
| 使用 Prisma 而非直接 SQL | 类型安全、迁移管理、多数据库兼容 |
| 开发用 SQLite，生产用 PostgreSQL | 降低开发环境搭建成本 |
| 前端 React.lazy 懒加载所有页面 | 优化首屏加载性能 |
| JWT 认证而非 Session | 无状态、易于扩展、支持多端 |
| 金额字段用 Decimal（生产PostgreSQL） | 避免浮点精度丢失 |
| 软删除而非物理删除 | 数据可恢复、合规审计 |
| 前端 API 代理到后端 4001 | 开发环境解决跨域问题 |
