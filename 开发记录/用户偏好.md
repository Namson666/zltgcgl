# 用户偏好

> 记录用户（Namson）的交互偏好和开发约束
> 编写时间：2026-05-17

---

## 一、交流语言

- **100% 使用中文交流**
- 代码注释可以使用英文，但解释和说明必须用中文
- 用户是技术小白，解释需通俗易懂，说明"为什么"这样做

---

## 二、代码修改原则

### 保守修复原则
- 修复问题时只修改必要的代码，不做无关改动
- 修改代码时必须考虑关联位置，不应只看一处改一处
- 优先使用 `Edit` 工具而非 `Write` 工具修改现有代码
- 绝不修改任何现有代码文件（除非用户明确要求）

### 安全第一
- 绝不写入密钥、密码、API Key 等敏感信息到代码中
- 绝不执行 `sudo npm` 命令
- 不跳过 git hooks（`--no-verify` 等）

---

## 三、Agent 角色分工

### 主 Agent（本 Agent）职责
- **只做分析和审核，不做大规模代码修改**
- 分析项目结构、理解需求
- 审查代码质量、发现潜在问题
- 创建开发记录和文档
- 执行备份和部署操作

### 开发 Agent 职责
- 具体的代码编写和修改
- Bug 修复和功能开发
- 由专门的开发 Agent 执行代码变更

---

## 四、数据管理原则

### 数据优先
- 数据安全第一，代码次之
- 敏感配置文件（.env）必须放在 .gitignore 中
- 数据库备份定期执行
- 所有业务数据采用软删除（isActive），不物理删除

### 数据隔离
- 三层数据隔离：合同 → 项目部 → 项目名称
- 多租户数据严格隔离
- 所有业务表必须包含 tenantId

---

## 五、调试与交付

### 自检验证原则
- 发现问题后：调试 → 修复 → 验证，三步完成
- 确认成功后再交付，不让用户反复测试
- 同一问题 3 次未解决，升级到 Cowork

### 交付标准
- 构建必须通过
- 相关测试必须通过
- 不留下调试代码（console.log 等）

---

## 六、项目特定偏好

1. **文件组织**：按功能/领域组织，不按文件类型
2. **API 响应格式**：统一使用 `{ success: boolean, data?: any, error?: string }`
3. **代码注释**：关键逻辑需要中文注释说明
4. **版本管理**：使用约定式提交（feat/fix/refactor/docs/test/chore/perf/ci）
5. **开发端口**：后端 4001，前端 5173

---

## 七、外部依赖配置

- AI Key 和 OCR Key 优先从数据库读取（开发者后台配置），环境变量作为兜底
- 用户自行在开发者后台配置 AI 和 OCR，不通过环境变量硬编码
- 百度 OCR 使用 accurate_basic（免费高精度版）
- AI 大模型支持 OpenAI 和 MiniMax 两种提供商

---

## 八、禁止事项速查

| 禁止 | 原因 |
|------|------|
| 修改现有代码文件 | 防止引入新 Bug |
| 写入密钥到代码 | 安全原则 |
| 执行 sudo npm | 安全风险 |
| 跳过 git hooks | 质量保证 |
| 保留 console.log | 生产代码清洁 |
| 物理删除业务数据 | 数据可恢复 |
| 使用英文交流 | 用户偏好 |
| 硬编码配置值 | 灵活性 |
